# 開発者用メモ (非公開)

## セットアップと実行 (ローカル開発)

本プロジェクトはDockerコンテナとして実行します。以下の手順でローカル開発環境をセットアップしてください。

### 1. 前提条件

*   Docker および Docker Compose がインストールされていること。

### 2. 環境変数の設定

プロジェクトのルートディレクトリに、`.env.example`というテンプレートファイルが用意されています。これをコピーして`.env`ファイルを作成します。

```bash
cp .env.example .env
```

作成した`.env`ファイルを開き、`YOUR_...`となっているプレースホルダー部分を、ご自身のFirebaseプロジェクト、SupabaseのDB接続情報、Gemini APIキーなどの実際の値に書き換えてください。

#### 【重要】本番環境とテスト環境の分離について

アプリケーションを安全に運用するため、本番環境とローカルのテスト環境では、異なるFirebaseプロジェクトおよびSupabaseデータベースを使用します。

*   **ローカル環境 (Dockerビルド時):**
    *   手元の `.env` ファイルの設定が読み込まれます。
    *   **Firebase:** `SupplyChainTest` プロジェクトに接続。
    *   **Supabase:** `SupplyChainTest` データベースに接続。
*   **本番環境 (Renderデプロイ時):**
    *   Git上のソースコードではなく、Renderダッシュボードの「Environment Variables」の設定が優先されます。
    *   **Firebase:** `SupplyChain` (本番用) に接続。
    *   **Supabase:** `SupplyChain` (本番用) に接続。

これにより、ローカルでの修正やテストが本番データに影響を与えないようになっています。

### 3. アプリケーションのビルドと起動

ターミナルで以下のコマンドを実行します。

```bash
docker-compose up --build
```

### 4. データベースのリセット

1.  `.env`ファイルに `DB_RESET=true` を追加します。
2.  `docker-compose up --build` を実行すると、テーブルが一度削除され、最新のマスターCSVからデータが再投入されます。
3.  完了後は必ず `DB_RESET=true` を削除してください。

## 開発とデプロイの運用 (ワークフロー)

### 開発時の重要な留意事項
スムーズな開発とデプロイのため、以下の手順を遵守してください。

1.  **作業開始前の最新化**:
    *   新しい機能開発（ブランチ作成）を始める前に、必ず `main` ブランチを最新状態に更新してください。
    *   コマンド例: `git checkout main && git pull origin main`
    *   これにより、GitHub上でのマージ競合（コンフリクト）を最小限に抑えることができます。
2.  **データベース（スキーマ）の更新**:
    *   新しいカラムの追加やテーブル構造の変更を行った場合、環境に反映させるために一度だけ `DB_RESET=true` を設定して起動する必要があります。
    *   **本番環境 (Render) への適用時**: Renderの環境設定で `DB_RESET=true` を一時的に追加し、デプロイ完了後に必ず削除してください。
3.  **環境変数の同期**:
    *   Gemini APIキーなど新しい環境変数が必要になった場合は、速やかに `.env.example` を更新し、チーム（または本番環境）へ共有してください。

### ブランチ戦略
*   **`main`**: 本番用。RenderのAuto Deploy対象。直接コミット禁止。
*   **`develop`**: 開発統合用。次期リリースのための安定板。
*   **`feature/xxx`**: 個別の機能開発用。**必ずここから作業を開始します。**
*   **`hotfix/xxx`**: 公開中の不具合修正用。

### 新機能開発の手順
1.  `main`（または `develop`）を最新にし、そこから `feature/xxx` ブランチを作成。
2.  ローカル環境（Docker + テスト用DB/Firebase）で実装・検証。
3.  GitHubへプッシュし、プルリクエストを作成。
4.  競合がないことを確認し、マージを実行（本番環境へ自動デプロイ）。
